model router
============

Написан на Go.

По сути это прокси, которая группирует модели (несколько одинаковых бэкендов под одним
именем) и контролирует количество одновременных запросов (потоков) на каждого потребителя.

Поддерживает OpenAI API и проксирует запросы на бэкенды sglang / vllm.


1. Термины
----------
- **Модель (group)** — логическое имя (`deepseek`, `qwen`), за которым стоит пул из
  нескольких URL одинаковых бэкендов.
- **Бэкенд (upstream)** — конкретный URL инстанса sglang/vllm внутри группы.
- **Клиент (consumer)** — потребитель API, идентифицируется по CN клиентского
  сертификата (mTLS).


2. Конфигурация
---------------

### 2.1 Флаги командной строки
| Флаг          | Назначение                       | Дефолт            |
|---------------|----------------------------------|-------------------|
| `--port`      | TCP-порт прослушивания           | `8080`            |
| `--host`      | Адрес прослушивания              | `0.0.0.0`         |
| `--log-level` | Уровень логирования              | `info`            |
| `--config`    | Путь к YAML-конфигу              | `./config.yaml`   |

- Допустимые `--log-level`: `debug`, `info`, `warn`, `error`. 
- Флаги имеют приоритет над одноимёнными значениями из конфига (если такие появятся).

### 2.2 Файл конфигурации (YAML)
```yaml
global:
  loadBalancing: random        # random | roundRobin | lessQueue
  requestTimeoutSecs: 600      # таймаут одного проксируемого запроса
  healthRetrySec: 5            # интервал опроса /health бэкендов
  queuePollSec: 5              # интервал опроса глубины очереди (для lessQueue)

models:
  deepseek:
    urls:
      - http://model1:3000
      - http://model2:3000
      - http://model3:3000
  qwen:
    requestTimeoutSecs: 300    # переопределяет global для этой модели
    urls:
      - http://model4:3000
      - http://model5:3000
      - http://model6:3000

clients:
  leha:
    cn: CI01929381
    concurrencyLimit:
      - model: deepseek
        maxParallel: 5
  semen:
    cn: CI01922362
    concurrencyLimit:
      - model: deepseek
        maxParallel: 1
      - model: qwen
        maxParallel: 5

tls:
  cert: /opt/certs/cert.pem    # серверный сертификат (входящие соединения)
  key: /opt/certs/key.pem      # приватный ключ сервера
  cacert: /opt/certs/cacert.pem# CA для проверки клиентских сертификатов (mTLS)
  verify: true                 # требовать и проверять клиентский сертификат
```

### 2.3 Правила приоритета настроек
- `requestTimeoutSecs` на уровне модели переопределяет значение из `global`.
- Параметры, заданные только в `global`, применяются ко всем моделям.

### 2.4 Валидация конфига при старте
- Дублирующиеся имена моделей/клиентов, дублирующиеся CN — ошибка старта.
- Пустой список `urls` у модели — ошибка старта.
- Ссылка в `concurrencyLimit` на несуществующую модель — ошибка старта.
- Неверный `loadBalancing` — ошибка старта.
- При невалидном конфиге сервис не стартует и пишет понятную ошибку.

### 2.5 Перезагрузка конфига 
- Дефолт: конфиг читается только при старте, hot-reload не требуется. 

### 2.6 Определение движка
- Для определения формата ручек сперва следует сходить по адресу модели и выяснить каким движком она поднята.


3. Идентификация и авторизация клиентов
---------------------------------------
- Клиент идентифицируется по **CN из клиентского TLS-сертификата** (mTLS, `tls.verify: true`).
- CN сопоставляется с записью в `clients`.
- Если сертификат отсутствует/невалиден при `verify: true` → `401`.
- Если CN не найден среди `clients` → `403`. 
- Если клиент запросил модель, для которой у него нет записи в `concurrencyLimit`
  → безлимитный доступ.


4. Маршрутизация запросов
-------------------------
- Имя модели берётся из поля `model` тела запроса OpenAI (`chat/completions`,
  `completions`, `embeddings`).
- Имя маппится на группу из секции `models`.
- Если запрошенной модели нет в конфиге или нет живых бэкендов → `404` / `503`.
- Внутри группы выбор конкретного бэкенда выполняет балансировщик (раздел 6).


5. Контроль конкурентности
--------------------------
- Для пары (клиент, модель) одновременно выполняется не более `maxParallel` запросов.
- Поведение при превышении лимита: 
  - Запрос **ставится в очередь ожидания** до освобождения слота, но не дольше
    `requestTimeoutSecs`.
  - Если слот не освободился в течение таймаута → `429 Too Many Requests`.
- Счётчик слотов уменьшается при любом завершении запроса (успех, ошибка, разрыв
  соединения клиентом).


6. Балансировка нагрузки
------------------------
- `random` — случайный выбор живого бэкенда.
- `roundRobin` — по кругу среди живых бэкендов.
- `lessQueue` — выбор бэкенда с наименьшей текущей очередью.
  - Раз в `queuePollSec` (дефолт 5с) опрашивается глубина очереди каждого бэкенда.
  - Источник метрики:
    - vllm: Prometheus-метрика `vllm:num_requests_waiting` с `/metrics`.
    - sglang: /get_server_info.
  - Между опросами используется последнее известное значение.
- Балансировщик выбирает только среди бэкендов в статусе healthy.
- Если живых бэкендов нет → `503`.


7. Health-checks
----------------
- Каждый бэкенд периодически (интервал `healthRetrySec`, дефолт 5с) опрашивается:
  - vllm → `GET /health`
  - sglang → `GET /health_generate`
- Бэкенд, не ответивший успехом, помечается unhealthy и исключается из балансировки.
- Если **все** бэкенды модели unhealthy, модель не показывается в `/v1/models`,
  а запросы к ней получают `503`.
- При восстановлении бэкенд возвращается в ротацию автоматически.


8. HTTP API
-----------
Проксируемые OpenAI-ручки:
- `POST /v1/chat/completions`
- `POST /v1/completions`
- `POST /v1/embeddings`
- `GET  /v1/models` — только модели, у которых есть хотя бы один healthy бэкенд.

Поддержка streaming (`"stream": true`, SSE) — да, ответ
проксируется потоково без буферизации.

Служебные ручки:
- `GET /docs` — отдаёт OpenAPI-спецификацию сервиса.
- `GET /metrics` — Prometheus-метрики сервиса.
- `GET /health` (или `/healthz`) — liveness самого роутера.


9. Метрики Prometheus (`/metrics`)
----------------------------------
Минимальный набор, можно расширить
- `router_requests_total{client,model,code}` — счётчик запросов.
- `router_request_duration_seconds{client,model}` — гистограмма латентности.
- `router_inflight_requests{client,model}` — текущая конкурентность.
- `router_queue_wait_seconds{client,model}` — время ожидания слота.
- `router_concurrency_rejected_total{client,model}` — отказы по лимиту (429).
- `router_backend_up{model,url}` — статус health бэкенда (1/0).
- `router_backend_queue_size{model,url}` — наблюдаемая глубина очереди (lessQueue).


10. Логирование
---------------
- Уровень управляется флагом `--log-level`.
- Формат логов: Plain Text
- При старте в лог выводится сводка:
  - какие модели сконфигурированы и какие из них сейчас healthy/unhealthy;
  - выбранный режим балансировки, таймауты;
  - список клиентов и их лимиты;
  - адрес/порт, статус TLS (включён mTLS или нет).
- Логировать: входящий запрос (client, model, путь), исход (код, длительность),
  ошибки проксирования и health-переходы бэкендов.


11. TLS
-------
- Серверный TLS на входящих соединениях (`cert` + `key`).
- При `verify: true` — mTLS: клиент обязан предъявить сертификат, подписанный CA
  из `cacert`; CN используется для идентификации клиента.
- При `verify: false` — обычный TLS без клиентских сертификатов. Тогда механизм
  идентификации клиента **[уточнить]** (например, заголовок) — по умолчанию все
  запросы считаются анонимными и отклоняются, т.к. лимиты привязаны к CN.
- Соединения к бэкендам — по схеме из их URL (http/https). **[уточнить]** нужна ли
  проверка сертификатов апстримов отдельной настройкой.


12. Надёжность и эксплуатация
-----------------------------
- Таймаут проксируемого запроса — `requestTimeoutSecs` (модель → global).
- Graceful shutdown по SIGTERM/SIGINT: дождаться завершения активных запросов
  (с разумным предельным временем). 
- Отмена контекста при разрыве соединения клиентом должна прерывать запрос к бэкенду.


13. Требования к коду и репозиторию
-----------------------------------
- Версия Go: 1.22+
- Архитектура: Package-by-Layer. Пример дерева:
  ```
  cmd/router/main.go          # точка входа, парсинг флагов
  internal/config/            # загрузка и валидация YAML
  internal/handler/           # HTTP-слой, OpenAI-ручки, /metrics, /docs
  internal/service/           # бизнес-логика: маршрутизация, лимиты
  internal/balancer/          # random / roundRobin / lessQueue
  internal/health/            # опрос health и очередей бэкендов
  internal/proxy/             # обратный прокси к апстримам (в т.ч. streaming)
  internal/metrics/           # Prometheus-метрики
  internal/auth/              # идентификация клиента по CN
  ```
- Линтер: `golangci-lint` с конфигом в репозитории; сборка проходит без ошибок линтера.
- Юнит-тесты на конфиг, балансировку, контроль конкурентности. 
- Артефакты сборки: `Makefile` и `Dockerfile`.
